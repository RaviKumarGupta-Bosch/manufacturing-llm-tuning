Step 1

What Is Model Tuning?

Understand why generic LLMs fail in specialized domains and how tuning fixes this.

🎯 The Problem: A generic model like llama3.2 has broad knowledge but no specialization. Ask it about CNC spindle vibration thresholds — it guesses. Ask a tuned model — it cites ISO 10816 zone limits to the millimeter per second and tells you to stop production.

Three Levels of Tuning

Level 1

⚙️ Modelfile Engineering

⏱ 5 minutes · No GPU

System prompt + few-shot examples + parameter tuning. Works immediately with any Ollama model.

What changes: Behaviour (not weights)
How: ollama create mfg-expert -f Modelfile
Best for: Format enforcement, domain vocabulary, structured outputs
Level 2

🔍 RAG

⏱ Hours · No GPU

Retrieval-Augmented Generation. Inject knowledge from knowledge_base.md at query time.

What changes: Context per request
How: Embed docs → vector store → retrieve → inject
Best for: Live knowledge, large doc sets, updatable facts
Level 3

🧠 LoRA Fine-Tuning

⏱ Hours–Days · GPU needed

Actual gradient-based weight updates using manufacturing_train.jsonl via Unsloth.

What changes: Model weights (adapter layers)
How: Unsloth → GGUF export → Ollama import
Best for: Deep domain fluency, implicit knowledge, style transfer
✅ This project demonstrates Level 1 — the fastest path to a domain-tuned model. Level 3 LoRA instructions are in Step 8 when you're ready to go deeper.

Step 3

Training Data Format

The quality of training data determines the quality of tuning. Learn the ChatML conversation format.

Format: One Conversation Per Line (JSONL)

Each line in manufacturing_train.jsonl is a complete JSON object — a conversation with three roles:

{"messages": [
  {
    "role": "system",
    "content": "You are ManuBot, a senior manufacturing engineer AI..."
  },
  {
    "role": "user",
    "content": "CNC spindle vibration jumped from 0.8 to 3.9 mm/s. What should I do?"
  },
  {
    "role": "assistant",
    "content": "**Root Cause:** Bearing inner race spalling...\n\n**Recommended Action:** 1. Stop spindle..."
  }
]}

Role Breakdown

RolePurposeKey Tip
system Sets the AI persona and constraints for all conversations Be specific — list exact skills, response format, knowledge areas
user The question or problem statement Vary phrasing — don't repeat the same question with different words
assistant The ideal answer the model should learn to reproduce Follow your desired format exactly — this is the gold standard

Good vs Bad Training Examples

❌ Bad Training Example
{"messages": [
  {"role": "system",
   "content": "You are a helper."},
  {"role": "user",
   "content": "OEE dropped?"},
  {"role": "assistant",
   "content": "OEE drop could be many things. Check the machine."}
]}
  • Vague system prompt — no domain context
  • No structured response format enforced
  • 9-word response — far too short
  • No standards or thresholds cited
✅ Good Training Example
{"messages": [
  {"role": "system",
   "content": "You are ManuBot, expert in OEE,
   FMEA, ISO 22400... Always respond:
   Root Cause / Action / Priority / Standard."},
  {"role": "user",
   "content": "OEE dropped 82%→61%. Availability 71%."},
  {"role": "assistant",
   "content": "**Root Cause:** Unplanned downtime...
   \n**Recommended Action:** 1. Run 5-Why...
   \n**Priority:** HIGH
   \n**Standard:** ISO 22400-2"}
]}
  • Rich system prompt with skills list
  • Structured format enforced in system prompt
  • Specific numbers (82%→61%, 71%)
  • Standard cited (ISO 22400-2)

Step 4

Modelfile Engineering

The Modelfile is Ollama's configuration format — a simple but powerful tuning mechanism that works instantly.

Side-by-Side: Base vs Tuned Modelfile

Modelfile.base — BEFORE
FROM llama3.2

PARAMETER temperature 0.7

SYSTEM """You are a helpful assistant."""
  • Generic identity — no manufacturing domain
  • High temperature → unpredictable answers
  • No response structure enforced
  • No domain vocabulary seeded
  • No examples → no output pattern
Modelfile.manufacturing — AFTER
FROM llama3.2

PARAMETER temperature 0.3
PARAMETER top_p 0.85
PARAMETER num_ctx 8192
PARAMETER repeat_penalty 1.1

SYSTEM """
You are ManuBot, a senior manufacturing
engineer AI with 20+ years of expertise
in OEE, FMEA, ISO 10816, AIAG-VDA FMEA...

Always structure responses as:
**Root Cause:** ...
**Recommended Action:** ...
**Priority:** HIGH / MEDIUM / LOW
**Relevant Standard:** ...
"""

MESSAGE user "Spindle at 3.9 mm/s"
MESSAGE assistant "**Root Cause:** ..."
  • Domain persona with explicit expertise list
  • Low temperature → precise, consistent answers
  • Larger context window for long structured responses
  • Exact output format enforced in system prompt
  • Few-shot MESSAGE pairs demonstrate the format

Key Parameters Explained

ParameterBaseTunedEffect
temperature0.70.3Lower = more deterministic. Critical for technical accuracy.
top_p—0.85Nucleus sampling — limits token selection to most probable.
num_ctx20488192Context window — allows longer structured multi-section responses.
repeat_penalty1.01.1Reduces repetitive phrases in long responses.
💡 The MESSAGE directive is Ollama's few-shot mechanism. Each MESSAGE user / MESSAGE assistant pair teaches the model exactly what a correct response looks like. More examples = stronger format adherence.

Step 9

PEFT, LoRA & QLoRA Fine-Tuning

Go beyond Modelfiles — use gradient-based training to deeply embed domain expertise into model weights.

The Fine-Tuning Landscape

Full Fine-Tuning updates every parameter (7B+), requiring 80–140 GB VRAM. PEFT (Parameter-Efficient Fine-Tuning) is an umbrella for methods that update only ~0.1–1% of parameters, reaching 90–95% of full fine-tuning quality at a fraction of the cost.
PEFT

Parameter-Efficient Fine-Tuning

Umbrella term covering LoRA, QLoRA, IA³, Prefix Tuning, and Prompt Tuning. Base weights stay frozen; only small adapter layers are trained.

  • ~0.1–1% of parameters trained
  • Adapters saved as tiny files (<200 MB)
  • Multiple adapters share one base model
LoRA

Low-Rank Adaptation

Inserts trainable low-rank matrices A and B alongside frozen transformer weights. After training, they merge back — zero extra inference latency.

  • Rank r = 4–64 (start with 16)
  • Alpha α = scaling factor (typically = r)
  • ~24–40 GB VRAM for a 7B model
QLoRA

Quantized LoRA

Loads the base model in 4-bit NF4 quantization (bitsandbytes), then applies LoRA adapters in full precision — the standard choice for consumer GPUs.

  • 7B model fits in 6–8 GB VRAM
  • NF4 quantization via bitsandbytes
  • ~5–8% quality loss vs full fine-tuning

How LoRA Works — The Math

For a frozen weight matrix W (d × d), LoRA trains two small matrices instead of updating all d² weights:

W′ = W + (α ÷ r) × B × A A ∈ ℝr×d  ·  B ∈ ℝd×r  —  where r ≪ d  (e.g. r=16, d=4096 → 131K params vs 16.7M)

Only A and B are trained. W stays frozen. After training, B×A is merged back into W so inference speed is identical to the original model.

Key Hyperparameters

rank (r)
Controls adapter expressiveness. r=4 is fast and lightweight; r=64 is richer. Recommended: r=16 for manufacturing domain (niche vocabulary + structured answer format).
alpha (α)
Scaling factor for LoRA updates. Start with α = r (e.g. α=16 when r=16). Higher α = stronger LoRA influence. Can cause overfitting on small datasets.
target_modules
Which transformer layers get LoRA adapters. For LLaMA models, use all projection layers: q_proj, v_proj, k_proj, o_proj, gate_proj, up_proj, down_proj.
